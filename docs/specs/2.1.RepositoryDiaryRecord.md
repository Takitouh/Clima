# 2.1 Repository DiaryRecord - Daily Weather Data

## Overview

Complete implementation of the daily weather repository layer. Handles CRUD operations and two main
use cases: database retrieval and API fetching with orchestration between them.

**Interface:** `data/repository/IRepositoryDiaryRecord.kt`

**Status:** ✅ COMPLETE

## Entity: DiaryRecord (Daily Weather)

```kotlin
@Entity(
  tableName = "diary_record",
    indices = [Index(value = ["date", "city", "region", "country"], unique = true)]
)
data class DiaryRecord(
    @PrimaryKey(autoGenerate = true)
    var id: Int = 0,
    var date: String,                 // YYYY-MM-DD (UTC date only)
    var city: String,                 // City name
    var region: String,               // State/Province
    var country: String,              // Country name
    var latitude: Double? = null,     // Device coordinates
    var longitude: Double? = null,    // Device coordinates
    @ColumnInfo(name = "temp_max_fahrenheit")
    var maxTempFahrenheit: Float,
    @ColumnInfo(name = "temp_min_fahrenheit")
    var minTempFahrenheit: Float,
    @ColumnInfo(name = "temp_max_celsius")
    var maxTempCelsius: Float,
    @ColumnInfo(name = "temp_min_celsius")
    var minTempCelsius: Float
)
```

**Unique Constraint:** `(date, city, region, country)`

- Only ONE record per day per location (city+region+country)
- Cannot have duplicate daily records for same location
- Ensures clean historical data without duplicates

## Repository Interface: IRepositoryDiaryRecord

### 1. obtainHistoricalWeatherFromDatabase()

**Pure database read operation** - No API fallback.

```kotlin
suspend fun obtainHistoricalWeatherFromDatabase(
    date: String,
    city: String,
    region: String,
    country: String
): WeatherRecordDto?
```

**Parameters:**

- `date`: Date in YYYY-MM-DD format
- `city`: City name (e.g., "New York")
- `region`: State/Province (e.g., "NY")
- `country`: Country name (e.g., "United States")

**Returns:**

- `WeatherRecordDto` if record found
- `null` if not in database

**Constraints:**

- ❌ Cannot be used for future dates (throws IllegalArgumentException)

**Error Handling:**

- ❌ If SecurityException or DB error → throws exception

**Use Cases:**

- Check if data is already cached
- Offline mode (no API calls)
- Manual cache validation
- Analytics queries

**Example:**

```kotlin
val weather = repository.obtainHistoricalWeatherFromDatabase(
    date = "2026-04-26",
    city = "New York",
    region = "NY",
    country = "United States"
)

if (weather != null) {
    println("Found in cache: ${weather.maxTempFahrenheit}°F")
} else {
    println("Not cached, need to fetch from API")
}
```

### 2. fetchHistoricalWeatherFromApi()

**Pure API fetch operation** - Automatically saves to database.

```kotlin
suspend fun fetchHistoricalWeatherFromApi(
    date: String,
    city: String,
    region: String,
    country: String
): WeatherRecordDto?
```

**Parameters:**

- Same as above

**Returns:**

- `WeatherRecordDto` with fetched weather data
- `null` if API returns no data

**Process:**

```
1. Validate date is not future
2. Call WeatherAPI with city and date
3. Parse API response
4. Auto-save to database
5. Return DTO (without ID)
```

**Constraints:**

- ❌ Cannot be used for future dates (throws IllegalArgumentException)

**Error Handling:**

- ❌ If API fails → throws exception
- ❌ If DB save fails → throws exception
- ❌ If date validation fails → throws IllegalArgumentException

**Use Cases:**

- Manual refresh (bypass cache)
- Get latest weather data
- Handle stale cache scenarios
- User explicit "Refresh" action

**Example:**

```kotlin
try {
    val weather = repository.fetchHistoricalWeatherFromApi(
        date = "2026-04-26",
        city = "New York",
        region = "NY",
        country = "United States"
    )
    println("Fetched: ${weather?.maxTempCelsius}°C")
} catch (e: IllegalArgumentException) {
    println("Cannot fetch future weather: ${e.message}")
} catch (e: Exception) {
    println("API error: ${e.message}")
}
```

### 3. obtainOrFetchHistoricalWeather() [ORCHESTRATOR]

**Smart combination** - Tries database first, falls back to API.

```kotlin
suspend fun obtainOrFetchHistoricalWeather(
    date: String,
    city: String,
    region: String,
    country: String
): WeatherRecordDto?
```

**Parameters:**

- Same as above

**Returns:**

- `WeatherRecordDto` from database OR API
- `null` if neither source has data

**Process:**

```
1. Validate date is not future
2. Try database query
   ├─ If found → return data ✅
   └─ If not found → continue
3. Fetch from API
4. Auto-save to database
5. Return fetched data ✅
```

**Constraints:**

- ❌ Cannot be used for future dates (throws IllegalArgumentException)

**Error Handling:**

- ❌ If date validation fails → throws IllegalArgumentException
- ❌ If both DB and API fail → throws exception
- Logs all steps for debugging

**Use Cases:**

- **PRIMARY USE CASE** - Main entry point for UI/ViewModel
- Get weather with smart caching
- Minimize API calls (use cache first)
- Automatic fallback when cache empty

**Example:**

```kotlin
// Recommended usage
val weather = repository.obtainOrFetchHistoricalWeather(
    date = today,
    city = locationInfo.city,
    region = locationInfo.region,
    country = locationInfo.country
)

if (weather != null) {
    // Display in UI
    updateUI(weather)
} else {
    // Show error
    showErrorScreen("Weather data unavailable")
}
```

**Architecture Diagram:**

```
obtainOrFetchHistoricalWeather()
        │
        ├─→ Validate date (not future)
        │
        ├─→ Try Database
        │   ├─ Found? Return data ✅
        │   └─ Not found? Continue
        │
        └─→ Try API
            ├─ Success? Save & return ✅
            └─ Failure? Throw exception
```

### Error Handling Strategy

| Error         | Cause                     | Response                 | Handling         |
|---------------|---------------------------|--------------------------|------------------|
| Future date   | User/system               | IllegalArgumentException | Throw exception  |
| API failure   | Network                   | Exception                | Throw exception  |
| DB failure    | Corruption/permission     | Exception                | Throws exception |
| No data found | Empty cache + API offline | null                     | Returns null     |

## Unique Constraint Behavior

### Example Scenario

```
Day 1 @ 10:00 AM - User opens app in NYC
├─ Record created: (2026-04-26, New York, NY, USA)
└─ HourlyRecord created for hour 10

Day 1 @ 3:00 PM - User opens app again in NYC
├─ Same day, same city → REUSE record
└─ HourlyRecord created for hour 15
(No new DiaryRecord record - constraint prevents duplicate)

Day 2 @ 10:00 AM - User opens app in NYC
├─ Different date → NEW record
├─ Record created: (2026-04-27, New York, NY, USA)
└─ HourlyRecord created for hour 10

Day 2 @ 1:00 PM - User opens app in Boston
├─ Different city → NEW record
├─ Record created: (2026-04-27, Boston, MA, USA)
└─ HourlyRecord created for hour 10
```

## HourlyRecord Relationship

DiaryRecord has **1-to-Many** relationship with HourlyRecord:

- One daily record → Multiple hourly records (one per hour)
- Hourly records reference parent via `diaryRecordId` (FK)
- Cascade delete (delete daily record → delete all hourly records)

See [2.2.RepositoryHourlyRecord.md](2.2.RepositoryHourlyRecord.md) for details.

## Performance Considerations

### Database Queries

- Query by unique constraint: `O(log n)` - indexed
- Full table scan: `O(n)` - only obtainAll()
- Index on (date, city, region, country) ensures fast lookups

### API Calls

- Cached in database after first fetch
- Only calls API if:
    1. Record not found in database AND
    2. obtainOrFetchHistoricalWeather() called
- Manual refresh via fetchHistoricalWeatherFromApi() always calls API
