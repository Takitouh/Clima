# 2.4 Repository PreferenciaUsuario - User Preferences Management

## Overview

Implementation of the user preferences repository layer. Handles CRU operations and preference
validation for managing user settings such as temperature unit preference (Celsius/Fahrenheit) and
time format (24-hour military or 12-hour AM/PM). Includes a singleton pattern for retrieving the
current user preference.

**Location:** `data/repository/impl/RepositoryPreferenciaUsuario.kt`

**Interface:** `data/repository/IRepositoryPreferenciaUsuario.kt`

**Status:** ✅ COMPLETE

## Entity: PreferenciaUsuario (User Preferences - Singleton)

```kotlin
@Entity(tableName = "preferencias_usuario")
data class PreferenciaUsuario(
    @PrimaryKey
    var id: Int = 1,  // FIXED to 1 - only one record per app (singleton pattern)
    @ColumnInfo(name = "medida_temp_fav")
    var medidaTempFav: String = "C",      // Temperature unit: "C" or "F"
    @ColumnInfo(name = "formato_militar")
    var formatoMilitar: Boolean = false   // Time format: true = 24h, false = 12h AM/PM
)
```

**Purpose:**

- Stores user application preferences as a singleton
- Persists settings across app sessions
- Enables personalized user experience
- Supports international preferences (temperature units, time formats)
- **SINGLETON PATTERN:** Only **ONE** record can exist per app (enforced at id=1)

**Singleton Pattern Design:**

- `id` is **fixed to 1** (NOT autoGenerated)
- Database primary key configuration prevents multiple records
- First app launch: Creates the single preference record (id=1)
- User updates: Same record is updated, never creates duplicates
- Always exactly one record in the table

**Fields:**

- `id`: Primary key (always = 1)
- `medidaTempFav`: Temperature measurement unit
    - `"C"` = Celsius (default)
    - `"F"` = Fahrenheit

- `formatoMilitar`: Time display format
    - `true` = Military/24-hour format (00:00-23:59)
    - `false` = 12-hour AM/PM format (12:00 AM - 11:59 PM)

## DAO: DaoPreferenciaUsuario

```kotlin
@Dao
interface DaoPreferenciaUsuario {
    /**
     * Insert or replace preference record.
     * With OnConflictStrategy.REPLACE, this overwrites the existing record at id=1.
     * Maintains singleton pattern - only one record exists.
     */
    @Insert(onConflict = OnConflictStrategy.REPLACE)
    suspend fun insertar(preferencia: PreferenciaUsuario): Long

    /**
     * Get the singleton preference record (always id=1).
     * Returns null if preferences not initialized yet.
     * This is the preferred method for singleton access.
     */
    @Query("SELECT * FROM preferencias_usuario WHERE id = 1")
    suspend fun obtenerPreferencia(): PreferenciaUsuario?
    
    /**
     * Update the singleton preference record.
     */
    @Update
    suspend fun actualizar(preferencia: PreferenciaUsuario): Int
}
```

**Key Methods:**

- `insertar()`: **NEW:** Uses `OnConflictStrategy.REPLACE` to enforce singleton by replacing record
  at id=1
- `obtenerPreferencia()`: **NEW:** Optimized query that directly fetches id=1 (singleton access)
- `actualizar()`: Updates existing preference record

**Singleton Strategy:**

The `REPLACE` conflict strategy ensures:

- First insert: Creates record at id=1
- Subsequent updates via `insert()`: Overwrites the existing record at id=1
- Database guarantees only one record exists

## Repository Interface: IRepositoryPreferenciaUsuario

### Method: insert()

**Creates or replaces the singleton preference record with validation.**

```kotlin
suspend fun insert(preferencia: PreferenciaUsuario): Long
```

**Parameters:**

- `preferencia`: PreferenciaUsuario object with preferences (id can be any value)

**Returns:**

- `Long`: ID of the inserted record (always = 1)

**Validation Rules:**

- Temperature unit must be "C" or "F"
- Format militar accepts any boolean value

**Singleton Enforcement:**

- Repository **automatically forces id=1** regardless of input
- If record already exists, REPLACE strategy **overwrites** the existing record
- Guarantees only one record can exist in database

**Error Handling:**

- Throws `IllegalArgumentException` if temperature unit is invalid
- Throws `Exception` if database operation fails

**Use Cases:**

- Initialize user preferences on first app launch
- Replace preferences when user makes changes (uses singleton pattern)

**Example:**

```kotlin
// First app launch - create default preferences
val defaultPreference = PreferenciaUsuario(
    medidaTempFav = "C",
    formatoMilitar = false
)
val prefId = repositoryPreferencia.insert(defaultPreference)
// Result: Preference saved with id=1

// Later - user changes preference (won't create new record)
val updatedPref = defaultPreference.copy(medidaTempFav = "F")
val replacedId = repositoryPreferencia.update(updatedPref)
// Result: id=1 record is replaced, no new record created
```

### Method: update()

**Modifies an existing preference record with validation.**

```kotlin
suspend fun update(preferencia: PreferenciaUsuario): Int
```

**Parameters:**

- `preferencia`: PreferenciaUsuario object with updated data (must have ID)

**Returns:**

- `Int`: Number of records updated (0 or 1)

**Validation Rules:**

- Temperature unit must be "C" or "F"
- Format militar accepts any boolean value

**Error Handling:**

- Throws `IllegalArgumentException` if temperature unit is invalid
- Throws `Exception` if database operation fails

**Important:**

- Preference must have valid ID from database
- Returns 0 if ID doesn't exist

**Use Cases:**

- Change temperature unit preference
- Toggle time format
- Update user settings from settings screen

**Example:**

```kotlin
val currentPref = repositoryPreferencia.obtainCurrentPreferencia()
if (currentPref != null) {
    // Switch from Celsius to Fahrenheit
    currentPref.medidaTempFav = "F"
    currentPref.formatoMilitar = true

    try {
        val updated = repositoryPreferencia.update(currentPref)
        Log.d("Preferences", "Updated $updated preference record(s)")
    } catch (e: IllegalArgumentException) {
        Log.e("Preferences", "Invalid value: ${e.message}")
    }
}
```

### Method: obtainCurrentPreferencia()

**Retrieves the current user preference (singleton pattern).**

```kotlin
suspend fun obtainCurrentPreferencia(): PreferenciaUsuario?
```

**Returns:**

- `PreferenciaUsuario`: Current preference if exists
- `null`: If no preference record found

**Behavior:**

- Implements singleton pattern (assumes one preference per user)
- Returns first record if multiple exist
- Returns null if database is empty

**Error Handling:**

- Throws `Exception` if database operation fails

**Use Cases:**

- Get current user settings during app lifecycle
- Check preferences before displaying weather data
- Initialize settings views

**Example:**

```kotlin
val currentPreference = repositoryPreferencia.obtainCurrentPreferencia()

if (currentPreference != null) {
    val tempUnit = currentPreference.medidaTempFav
    val timeFormat = if (currentPreference.formatoMilitar) "24h" else "12h AM/PM"

    println("Using $tempUnit for temperature, $timeFormat time format")
} else {
    println("No preferences set, using defaults")
    // Create default preference
    val defaultPref = PreferenciaUsuario(
        medidaTempFav = "C",
        formatoMilitar = false
    )
    repositoryPreferencia.insert(defaultPref)
}
```

**Dependencies:**

### Validation

**Temperature Unit Validation**

```kotlin
private fun validateTemperatureUnit(medidaTempFav: String) {
    if (medidaTempFav != "C" && medidaTempFav != "F") {
        throw IllegalArgumentException(
            "Invalid temperature unit: $medidaTempFav. Must be 'C' or 'F'"
        )
    }
}
```

Validation occurs before database operations to prevent invalid data:

- Called in `insert()` method
- Called in `update()` method

### Logging

All operations logged with TAG = "RepositoryPreferenciaUsuario":

- **DEBUG**: Operation start, success, preference values
- **ERROR**: Operation failures with exception details

### Coroutine Context

All database operations run on `Dispatchers.IO` for non-blocking database access.

## Singleton Pattern Design

The repository implements a singleton preference pattern:

```
One User → One Active Preference Record → One Settings State
```

**Design Rationale:**

- Most apps maintain ONE active user at a time
- Simplified preference retrieval via `obtainCurrentPreferencia()`
- Future enhancement: Multi-user support can expand this

**Initialization Flow:**

```
App Launch
    ↓
obtainCurrentPreferencia()
    ↓
    ├─ Preference exists? → Use it
    ├─ Preference missing? → Create default
    └─ Multiple records? → Use first, log warning
```

## Error Handling

| Error Type                 | Cause                             | Response                 | Recovery                       |
|----------------------------|-----------------------------------|--------------------------|--------------------------------|
| Invalid temperature unit   | User input "X" instead of "C"/"F" | IllegalArgumentException | Validate input before save     |
| Duplicate record insert    | Multiple preferences exist        | Exception                | Use obtainCurrentPreferencia() |
| Database operation timeout | DB locked or slow                 | Exception                | Retry with backoff             |
| SQL constraint violation   | Data integrity issue              | Exception                | Reset preferences              |

        super.onCreate(savedInstanceState)
