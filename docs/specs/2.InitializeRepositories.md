## Key Features

### ✅ 1. Complete English Refactoring

All field names converted from Spanish to English:

- `fecha` → `date`
- `ciudad` → `city` (part of new composite key)
- `ubicacion` → Replaced with `city + region + country`
- `pais` → `country` (new field)
- `region` → `region` (new field for State/Province)

### ✅ 2. Location Model: City + Region + Country

Unique identification ensures no ambiguity:

- Same city name can exist in multiple countries
- Same city can exist in multiple regions within a country
- Composite unique constraint: `(date, city, region, country)`

**Example:**

```
❌ "New York" alone = ambiguous (USA? Canada? Multiple US states?)
✅ "New York, NY, USA" = unique (Manhattan Island)
✅ "New York, ON, Canada" = unique (different city)
```

### ✅ 3. Coordinate Storage

Stores device coordinates for:

- Future map features (pinpoint on map)
- Audit trail (where was user when record was created)
- Foundation for multi-location support

### ✅ 4. Three-Method Repository Pattern

Each method has distinct responsibility:

1. **obtainHistoricalWeatherFromDatabase()** - Pure DB read
2. **fetchHistoricalWeatherFromApi()** - Pure API fetch + DB save
3. **obtainOrFetchHistoricalWeather()** - Orchestrator (DB→API fallback)

### ✅ 5. Data Transfer Objects (DTOs)

Clean separation between layers:

- `WeatherRecordDto` - Business logic layer (no ID)
- `LocationInfo` - Geocoding results
- `WeatherRecord` - Database entity (with ID)

## Database Schema Changes

### Old Schema

```
weather_record (date, location) UNIQUE
├── date: String
├── location: String (ambiguous - city only)
└── temperatures
```

### New Schema

```
diary_record (date, city, region, country) UNIQUE  ✅
├── id: Int (PK)
├── date: String (YYYY-MM-DD UTC)
├── city: String (e.g., "New York")
├── region: String (e.g., "NY")
├── country: String (e.g., "United States")
├── latitude: Double? (for maps)
├── longitude: Double? (for maps)
└── temperatures (separated Fahrenheit/Celsius)

hourly_record CLEANED UP  ✅
├── id: Int (PK)
├── diary_record_id: Int (FK)
├── hour: Int (0-23)
├── temp_fahrenheit: Float
├── temp_celsius: Float
(Removed redundant: date, location)
```

## API Integration

### WeatherAPI Endpoint

```
GET https://api.weatherapi.com/v1/history.json?key=API_KEY&q=CITY_NAME&dt=DATE
```

**Parameters:**

- `key`: API key (from BuildConfig.API_KEY)
- `q`: City name
- `dt`: Date in YYYY-MM-DD format

**Example:**

```
https://api.weatherapi.com/v1/history.json?key=abc123&q=New%20York&dt=2026-04-27
```

## Validation Rules

### ⚠️ CRITICAL: Future Date Validation

All historical weather methods CANNOT be used for future dates:

- `fetchHistoricalWeatherFromApi()` throws `IllegalArgumentException` if date is future
- `obtainOrFetchHistoricalWeather()` throws `IllegalArgumentException` if date is future
- Validated via `TimeUtil.isValidHistoricalDate()`
- Prevents API calls for dates that don't exist yet

## Error Handling Strategy

| Error               | Location     | Response                        | User Sees                       |
|---------------------|--------------|---------------------------------|---------------------------------|
| Permission denied   | LocationUtil | Throws SecurityException        | "Location couldn't be obtained" |
| Location disabled   | LocationUtil | Returns null                    | "Location couldn't be obtained" |
| No geocoding result | LocationUtil | Returns null                    | "Location couldn't be obtained" |
| API call fails      | Repository   | Throws exception                | "Weather couldn't be obtained"  |
| Database error      | Repository   | Throws exception                | "Weather couldn't be obtained"  |
| Future date         | Repository   | Throws IllegalArgumentException | Error in logs                   |
